# Master Glossary

This glossary defines every technical term used in the book. Entries are listed alphabetically. Each entry includes a plain-English explanation and a reference to where the concept goes deeper.

---

## Anti-enumeration

A design technique that prevents an attacker from using an API's response to discover which email addresses, usernames, or other identifiers are registered in a system. For example, returning the same error message for "user not found" and "wrong password" prevents someone from probing an email list to find which ones have accounts. This project applies anti-enumeration on the OTP send endpoint, the auth options endpoint, and the recovery endpoints.

*See also:* [OWASP Authentication Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Authentication_Cheat_Sheet.html)

---

## Assertion

In WebAuthn terminology, an assertion is the authentication-side response: a signed JSON structure produced by the authenticator that proves possession of the private key. The authenticator signs the challenge and some authenticator data with the stored private key. The server verifies this signature with the stored public key. Contrast with [Attestation](#attestation) (the registration-side credential verification).

*See also:* [W3C WebAuthn §6.3 Authenticator Data](https://www.w3.org/TR/webauthn-3/#sctn-authenticator-data)

---

## Attestation

In WebAuthn, attestation is the mechanism by which the authenticator proves (during registration) that it is a legitimate, trusted device. The authenticator provides a certificate chain rooted in the device manufacturer. The server can verify this against the FIDO Metadata Service to confirm the authenticator is genuine. Attestation can be `none` (no proof), `indirect`, `direct`, or `enterprise`. This project accepts all attestation types.

*See also:* [FIDO Metadata Service](https://fidoalliance.org/metadata/)

---

## Authenticator

In the WebAuthn context, an authenticator is the hardware or software entity that holds private keys and performs the signing operations. Authenticators come in two types: [Platform Authenticators](#platform-authenticator) (built into the device) and [Cross-Platform Authenticators](#cross-platform-authenticator) (external devices like hardware security keys). Examples: iPhone Face ID (platform), YubiKey (cross-platform).

*See also:* [W3C WebAuthn §5.1](https://www.w3.org/TR/webauthn-3/#authenticator)

---

## Binding

In the Cloudflare Workers context, a binding is a configuration that gives a Worker access to a resource — a D1 database, a KV namespace, a Durable Object class, or an ASSETS directory. Bindings are declared in `wrangler.toml` and accessible in Worker code as properties of the `env` parameter. Example: `env.AUTH_KV` (KV binding), `env.varun_portfolio_auth` (D1 binding).

*See also:* [Cloudflare Workers Bindings](https://developers.cloudflare.com/workers/runtime-apis/bindings/)

---

## Browser Autofill UI

The browser's built-in dropdown that appears when a user focuses an input field with a relevant `autocomplete` attribute. Normally shows saved passwords, emails, or addresses. With WebAuthn conditional mediation and `autocomplete="username webauthn"`, the browser augments this dropdown to also show passkeys registered for the current site. The user can tap a passkey entry to sign in without clicking a separate button.

*See also:* [Conditional Mediation](#conditional-mediation)

---

## CBOR

Concise Binary Object Representation (CBOR, RFC 7049) is a binary data serialization format designed to be compact and efficient. WebAuthn uses CBOR extensively in the authenticator data structures, including encoding public keys in COSE format. The `@simplewebauthn/server` library handles all CBOR parsing internally; application developers typically don't interact with CBOR directly.

*See also:* [RFC 7049](https://www.rfc-editor.org/rfc/rfc7049)

---

## Challenge

In WebAuthn, a challenge is a cryptographically random value (typically 32 bytes) generated by the server and included in the authentication options sent to the browser. The authenticator signs the challenge as part of the assertion. The server verifies that the signed challenge matches what was issued. Challenges are single-use and short-lived (60–120 seconds in this project) to prevent replay attacks.

*See also:* [W3C WebAuthn §13.4.3 Verifying an Authentication Assertion](https://www.w3.org/TR/webauthn-3/#sctn-verifying-assertion)

---

## Compatibility Date

A Cloudflare Workers configuration setting (`compatibility_date` in `wrangler.toml`) that specifies which version of the Workers runtime API the Worker expects. Cloudflare gates breaking runtime changes behind compatibility dates — a Worker with an older date continues to see the old behavior. Setting a recent date opts into all improvements up to that date.

*See also:* [Cloudflare Compatibility Dates](https://developers.cloudflare.com/workers/configuration/compatibility-dates/)

---

## Conditional Mediation

A WebAuthn feature (`mediation: 'conditional'` in `navigator.credentials.get()`) that allows the browser to present passkeys in its standard autofill dropdown without showing a modal dialog. The credential request is "conditional" — it resolves only if the user picks a passkey from the autocomplete. This requires an input element with `autocomplete="username webauthn"` and the server to have pre-loaded authentication options. See [Chapter 5](../chapters/05-passkeys-and-webauthn.md) for the full implementation.

*See also:* [WebAuthn Conditional Mediation explainer](https://github.com/w3c/webauthn/wiki/Explainer:-WebAuthn-Conditional-UI)

---

## Contract Ratio

See [WCAG](#wcag).

---

## Cookie

An HTTP cookie is a small piece of data set by the server (`Set-Cookie` header) and stored by the browser. The browser automatically includes cookies matching the domain/path on all subsequent requests (`Cookie` header). Session cookies are the most common use case. Security attributes: [HttpOnly](#httponly) (no JS access), [Secure](#secure-flag) (HTTPS only), [SameSite](#samesite) (cross-site protection).

*See also:* [MDN: HTTP cookies](https://developer.mozilla.org/en-US/docs/Web/HTTP/Cookies)

---

## CORS

Cross-Origin Resource Sharing. A browser security mechanism that controls whether JavaScript on one origin (scheme + host + port) can read responses from a different origin. Without CORS headers, a `fetch()` from `https://varunr.dev` to `https://varunr.dev/api/auth/me` would be allowed (same origin), but a fetch from `http://localhost:5173` to `http://localhost:8787/api/auth/me` would be blocked (different ports = different origins). The `Access-Control-Allow-Origin` header tells the browser which origins are allowed.

*See also:* [MDN: CORS](https://developer.mozilla.org/en-US/docs/Web/HTTP/CORS)

---

## COSE

CBOR Object Signing and Encryption (RFC 8152) is the format used by WebAuthn to encode public keys. A COSE key encodes the algorithm identifier, key type, and key material in a CBOR structure. The `@simplewebauthn/server` library handles COSE encoding and decoding. This project stores public keys as base64url-encoded COSE blobs in the `passkey_creds.public_key` column.

*See also:* [RFC 8152](https://www.rfc-editor.org/rfc/rfc8152)

---

## Counter

See [Sign Count](#sign-count).

---

## Credential ID

The unique identifier for a WebAuthn credential, assigned by the authenticator during registration. It is a variable-length byte array, typically base64url-encoded. The credential ID is stored in `passkey_creds.id` and used as the primary key for passkey lookup during authentication. When the browser authenticates, it includes the credential ID in the response so the server knows which public key to use for verification.

*See also:* [W3C WebAuthn §5.4.2](https://www.w3.org/TR/webauthn-3/#dom-publickeycredential-rawid)

---

## Cross-Platform Authenticator

A WebAuthn authenticator that is separate from the device — typically a hardware security key (YubiKey, Google Titan Key, etc.) connected via USB, NFC, or Bluetooth. Cross-platform authenticators are not tied to a specific device and can be used on any machine. The credential may or may not be synced across devices depending on the device type. Contrast with [Platform Authenticator](#platform-authenticator).

*See also:* [FIDO2 Technical Overview](https://fidoalliance.org/fido2/fido2-web-authentication-webauthn/)

---

## D1 (SQLite)

Cloudflare's managed SQLite database service for Workers. D1 provides standard SQL, prepared statements, indexes, and foreign key support. It is strongly consistent within a single statement but not globally distributed like KV. D1 is ideal for persistent structured data that needs to be queried, joined, or sorted. In this project, D1 stores users, passkey credentials, sessions, recovery codes, and security events.

*See also:* [Cloudflare D1 Documentation](https://developers.cloudflare.com/d1/)

---

## Damped Oscillation

A physics concept describing a vibration that decreases in amplitude over time due to friction or resistance. In the logo animation (`Nav.jsx`), the Y-axis bounce uses a simplified damped sine: `(1 - phase) × sin(phase × 2π) × -5`. The `(1 - phase)` term is the damping factor — it linearly reduces the amplitude to zero as the symbol's display phase progresses. A fully damped oscillation settles to zero displacement.

*See also:* [Khan Academy: Damped Oscillations](https://www.khanacademy.org/science/physics/mechanical-waves-and-sound)

---

## Discoverable Credential

A WebAuthn passkey stored in the authenticator's "resident key" slot, meaning the authenticator can return it without being told the credential ID. This is required for conditional mediation — the browser needs to find passkeys for a site without knowing in advance which user is signing in. Requested with `residentKey: 'preferred'` in registration options. Most modern platform authenticators support discoverable credentials.

*See also:* [W3C WebAuthn §4 Terminology](https://www.w3.org/TR/webauthn-3/#client-side-discoverable-credential)

---

## Durable Object

A Cloudflare Workers primitive that provides a stateful, single-instance actor. Each Durable Object instance has its own JavaScript environment, in-memory state, storage (SQLite-backed), and can hold WebSocket connections open indefinitely. DO instances are addressed by name or ID and are guaranteed to be a single instance globally (no concurrency across instances for the same ID). This project uses one DO instance per user for the number-matching WebSocket broker.

*See also:* [Cloudflare Durable Objects](https://developers.cloudflare.com/durable-objects/)

---

## Edge Computing

Running computation at network edge locations (datacenters distributed globally, close to end users) rather than in a centralized cloud region. Cloudflare Workers is an edge computing platform — a Worker runs at whichever of Cloudflare's 300+ datacenters is closest to the requesting user. This reduces latency and eliminates the need to manage server infrastructure.

*See also:* [Cloudflare Workers](https://workers.cloudflare.com/)

---

## Golden Ratio

The mathematical constant φ ≈ 1.6180339887, derived from the ratio `a/b = (a+b)/a` where this ratio equals itself. It appears throughout nature and is used in design for aesthetically pleasing proportions. In the logo animation, the cycle length is `BASE_MS × PHI` (520ms × 1.618 ≈ 841ms). Because PHI is irrational, the animation never falls into a repeating pattern aligned with the browser's frame rate.

*See also:* [Wikipedia: Golden ratio](https://en.wikipedia.org/wiki/Golden_ratio)

---

## Hash

A function that maps data of arbitrary length to a fixed-size output (the hash or digest). Cryptographic hash functions are one-way (you cannot reverse a hash to find the input) and collision-resistant (it is computationally infeasible to find two inputs with the same hash). This project uses [SHA-256](#sha-256) for session token hashing and [PBKDF2](#pbkdf2) (which builds on SHA-256) for recovery code hashing.

*See also:* [MDN: SubtleCrypto.digest()](https://developer.mozilla.org/en-US/docs/Web/API/SubtleCrypto/digest)

---

## HOTP

HMAC-Based One-Time Password (RFC 4226). A one-time password algorithm where codes are generated from a shared secret and a counter value using HMAC-SHA1. The counter increments on each use. [TOTP](#totp) is built on top of HOTP, replacing the counter with a time-based value. This project does not implement HOTP, but it is the foundation of the TOTP system listed as future work.

*See also:* [RFC 4226](https://www.rfc-editor.org/rfc/rfc4226)

---

## HttpOnly

A cookie attribute that prevents JavaScript from reading the cookie via `document.cookie`. An `HttpOnly` cookie can only be sent by the browser as an HTTP header — it is invisible to client-side scripts. This blocks XSS (Cross-Site Scripting) attacks that attempt to steal session cookies by injecting JavaScript into the page. All session cookies in this project use `HttpOnly`.

*See also:* [MDN: Set-Cookie](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Set-Cookie#httponly)

---

## Identicon

A deterministic visual hash of an identifier (typically a username or email), used as an avatar for users who have not uploaded a profile picture. GitHub popularized the 5×5 symmetric grid identicon style. This project's `Identicon` component (`Nav.jsx`) generates identicons from email addresses using a seeded random number generator, ensuring the same email always produces the same avatar. WCAG contrast enforcement ensures visibility.

*See also:* [Wikipedia: Identicon](https://en.wikipedia.org/wiki/Identicon)

---

## isoBase64URL

A utility from `@simplewebauthn/server/helpers` for encoding and decoding base64url-encoded data. Base64url is a URL-safe variant of base64 that uses `-` and `_` instead of `+` and `/`. WebAuthn uses base64url encoding for credential IDs, challenges, and public keys. `isoBase64URL.fromBuffer(bytes)` converts binary data to a base64url string. `isoBase64URL.toBuffer(str)` reverses it.

*See also:* [RFC 4648 §5](https://www.rfc-editor.org/rfc/rfc4648#section-5)

---

## JWT

JSON Web Token (RFC 7519). A self-contained token format that encodes claims (like user ID and expiration) and optionally signs or encrypts them. JWTs are stateless — the server can verify a JWT without a database lookup. This project intentionally **does not** use JWTs for sessions. The reason: JWTs cannot be revoked without a blocklist. If a JWT session is stolen, it remains valid until expiry. This project uses opaque session tokens backed by KV, which can be revoked immediately.

*See also:* [RFC 7519](https://www.rfc-editor.org/rfc/rfc7519)

---

## KV Store

Cloudflare Workers KV is a globally-distributed key-value store. KV supports `get`, `put`, and `delete` operations with optional TTL (time-to-live). KV is eventually consistent — writes propagate to all edge locations within seconds, not milliseconds. KV is fast for reads (served from edge cache) but not suitable for queries, sorting, or counting. In this project, KV stores ephemeral data: OTP codes, WebAuthn challenges, session tokens, and number-match state.

*See also:* [Cloudflare KV](https://developers.cloudflare.com/kv/)

---

## Manifest

The Web App Manifest (`manifest.json` or similar) is a JSON file that describes how a web app should behave when installed as a [PWA](#pwa). It specifies the app name, icons, theme color, display mode, and start URL. In this project, the manifest is generated by `vite-plugin-pwa` during the build and includes 192×192 and 512×512 icons plus a maskable icon for Android adaptive icons.

*See also:* [MDN: Web App Manifests](https://developer.mozilla.org/en-US/docs/Web/Manifest)

---

## Miniflare

A local simulator for the Cloudflare Workers runtime, used for development and testing. It runs Workers locally without deploying to Cloudflare, simulating KV, D1, Durable Objects, and other bindings. Wrangler dev uses Miniflare internally when running `wrangler dev`.

*See also:* [Cloudflare Miniflare](https://miniflare.dev/)

---

## Number Matching

An out-of-band device approval mechanism that prompts a user to verify a short code displayed on two devices simultaneously. When a new device attempts sign-in, it shows a 2-digit number. The user's trusted device (already signed in) receives an approval request showing the same number. The user confirms the numbers match and approves the sign-in. This adds a human-in-the-loop second factor without SMS or TOTP. See [Chapter 8](../chapters/08-number-matching.md).

*See also:* [FIDO Alliance: Number Matching for Push Notifications](https://fidoalliance.org/number-matching-for-push-notification-mfa/)

---

## OTP

One-Time Password. A code that is valid for a single use. This project uses 6-digit numeric OTPs for email verification (during registration and account recovery). OTPs are generated with `crypto.getRandomValues`, stored in KV with a 10-minute TTL, and marked as `used: true` on first successful verification. Contrast with [TOTP](#totp) (time-based) and [HOTP](#hotp) (counter-based).

*See also:* [Wikipedia: One-time password](https://en.wikipedia.org/wiki/One-time_password)

---

## Passkey

A consumer-friendly term for a WebAuthn discoverable credential backed by a platform service (iCloud Keychain, Google Password Manager). Passkeys are designed to be synced across a user's devices and to replace passwords entirely. They use asymmetric cryptography: the private key never leaves the device (or the cloud backup), and the server stores only the public key. See [Chapter 5](../chapters/05-passkeys-and-webauthn.md).

*See also:* [FIDO Alliance: Passkeys](https://fidoalliance.org/passkeys/)

---

## PBKDF2

Password-Based Key Derivation Function 2 (RFC 8018). A key derivation function that applies a hash function (SHA-256 in this project) to an input repeatedly (iterated hashing) with a random salt to make brute-force attacks expensive. The output is a fixed-length key. Used in this project for recovery code hashing. The iteration count is 100 (POC-level) rather than the NIST-recommended ≥600,000 for passwords, due to Workers CPU limits. See [Chapter 9](../chapters/09-recovery-system.md) for the tradeoff analysis.

*See also:* [RFC 8018](https://www.rfc-editor.org/rfc/rfc8018)

---

## Platform Authenticator

A WebAuthn authenticator built into the operating system and hardware of the device, using the device's biometric or PIN system (Touch ID, Face ID, Windows Hello, Android biometrics). Platform authenticators are the most common type for passkeys and are what most users encounter when they "register a passkey" on their phone or laptop. They are accessed via the browser's native credential management API. Contrast with [Cross-Platform Authenticator](#cross-platform-authenticator).

*See also:* [FIDO2: Platform Authenticators](https://fidoalliance.org/fido2/)

---

## Preflight

An HTTP OPTIONS request sent by the browser before certain cross-origin requests, asking the server to confirm it accepts the actual request. Browsers send preflights for non-simple requests (POST with JSON body, custom headers, etc.) to prevent CORS bypasses. The server must respond with `Access-Control-Allow-Origin`, `Access-Control-Allow-Methods`, and `Access-Control-Allow-Headers`. This project handles preflights in `worker/index.js` with a `204 No Content` response.

*See also:* [MDN: Preflight request](https://developer.mozilla.org/en-US/docs/Glossary/Preflight_request)

---

## Public Key Credential

The W3C API object returned by `navigator.credentials.create()` (registration) or `navigator.credentials.get()` (authentication). It wraps the authenticator's response and provides properties like `rawId` (credential ID), `response` (the authenticator data + signature), and `type: 'public-key'`. The `@simplewebauthn/browser` library wraps this API and handles encoding.

*See also:* [MDN: PublicKeyCredential](https://developer.mozilla.org/en-US/docs/Web/API/PublicKeyCredential)

---

## Pub/Sub

Publish-Subscribe. A messaging pattern where publishers emit events without knowing which subscribers will receive them, and subscribers receive events without knowing which publisher sent them. In this project, the `NumMatchDO` Durable Object acts as a pub/sub broker: the Worker publishes approval requests to the DO via an internal HTTP call, and the DO delivers them to all subscribed trusted WebSocket clients.

---

## PWA

Progressive Web App. A web application that uses modern browser APIs to provide app-like behavior: offline support (via service worker), installability (via Web App Manifest), push notifications, and background sync. This project is configured as a PWA through `vite-plugin-pwa`, with a manifest, icons, and a Workbox-generated service worker that caches assets for offline use.

*See also:* [MDN: Progressive Web Apps](https://developer.mozilla.org/en-US/docs/Web/Progressive_web_apps)

---

## rAF

`requestAnimationFrame` — a browser API that schedules a callback to run before the next repaint. Using rAF for animations ensures they run at the display's refresh rate (typically 60fps) and are paused when the tab is hidden. This project's logo animation and the ParticleField background both use rAF directly (no animation library). See [Chapter 11](../chapters/11-frontend-architecture.md) for the math.

*See also:* [MDN: requestAnimationFrame](https://developer.mozilla.org/en-US/docs/Web/API/window/requestAnimationFrame)

---

## Rate Limiting

A mechanism that caps the number of requests a client can make in a time window. This project implements email-scoped rate limits: OTP sends (3/10min), full recovery attempts (3/10min), backup sign-in attempts (3/10min). Rate limit counters are stored in KV with TTLs matching the window. An IP-based rate limit layer (using `CF-Connecting-IP`) is noted as future work.

---

## React

A JavaScript library for building user interfaces using a component model. Components declare what the UI should look like given a state, and React handles updating the DOM when state changes. This project uses React 18 (with concurrent features via `createRoot`) and React Router v6 for client-side routing.

*See also:* [React Documentation](https://react.dev/)

---

## Relying Party (RP)

In WebAuthn, the Relying Party is the web application that is authenticating the user — in this case, `varunr.dev`. The RP is identified by its [RPID](#rpid). The RP generates challenges, stores public keys, and verifies assertions. The term comes from the FIDO2 spec and reflects the fact that the application "relies on" the authenticator to perform user verification.

*See also:* [W3C WebAuthn §4: Terminology](https://www.w3.org/TR/webauthn-3/#relying-party)

---

## Resident Key

See [Discoverable Credential](#discoverable-credential). "Resident key" is the original WebAuthn spec term; "discoverable credential" is the newer preferred term. Both refer to a passkey stored in the authenticator's permanent storage, retrievable without the credential ID.

---

## RPID

The Relying Party ID. A domain name string that identifies the WebAuthn relying party. Must be the effective domain or a registrable domain suffix of the origin. For this project: `varunr.dev`. Used in `generateRegistrationOptions`, `verifyRegistrationResponse`, `generateAuthenticationOptions`, and `verifyAuthenticationResponse`. **Changing the RPID after users have registered passkeys renders all existing passkeys unusable.**

*See also:* [W3C WebAuthn §4: rpId](https://www.w3.org/TR/webauthn-3/#rp-id)

---

## Salt

A random value mixed into a hash function to prevent pre-computation attacks (rainbow tables). Without a salt, an attacker could pre-compute hashes for all possible values in a space. With a per-item salt, every hash is unique even for identical inputs. This project uses 16-byte random salts for each recovery code and stores them alongside the hash in `recovery_codes.code_salt`.

*See also:* [Wikipedia: Salt (cryptography)](https://en.wikipedia.org/wiki/Salt_(cryptography))

---

## SameSite

A cookie attribute controlling when the cookie is sent on cross-site requests. Three values:
- `Strict` — cookie only sent on same-site requests (blocks CSRF)
- `Lax` — cookie sent on top-level navigation and same-site requests
- `None` — cookie sent on all requests (requires `Secure`)

This project uses `SameSite=Strict` for the session cookie, providing CSRF protection without an explicit CSRF token.

*See also:* [MDN: SameSite cookies](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Set-Cookie/SameSite)

---

## Secure Flag

A cookie attribute that restricts the cookie to HTTPS connections only. A cookie with the `Secure` flag is never sent over HTTP, preventing interception by network attackers. This project's session cookie uses `Secure`.

*See also:* [MDN: Set-Cookie Secure](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Set-Cookie#secure)

---

## Service Worker

A JavaScript file that runs in a background thread in the browser, separate from the page. Service workers can intercept network requests and serve responses from a cache — enabling offline functionality. This project uses a service worker generated by [Workbox](#workbox) via `vite-plugin-pwa`. It pre-caches all static assets (JS, CSS, HTML, fonts, images) and applies a CacheFirst strategy for Google Fonts.

*See also:* [MDN: Service Worker API](https://developer.mozilla.org/en-US/docs/Web/API/Service_Worker_API)

---

## Session Token

An opaque random value stored in a cookie that identifies an authenticated session. "Opaque" means the token has no inherent meaning — it is just a random string. The server looks it up in a database (or KV) to find the associated session data. Contrast with [JWT](#jwt), which is self-describing. This project generates session tokens as two concatenated UUIDs (64 hex chars, 128 bits of entropy).

---

## SHA-256

A cryptographic hash function from the SHA-2 family. Produces a 256-bit (32-byte) hash from any input. Used in this project via `crypto.subtle.digest('SHA-256', ...)` for session token hashing. SHA-256 is collision-resistant and pre-image resistant — properties needed for secure token hashing.

*See also:* [MDN: SubtleCrypto.digest()](https://developer.mozilla.org/en-US/docs/Web/API/SubtleCrypto/digest)

---

## Sign Count

An integer counter maintained by a WebAuthn authenticator, incremented each time the credential is used for authentication. The server stores the last-seen counter and compares it on the next use. If the incoming counter is not greater than the stored counter (and the counter is non-zero), it may indicate the credential has been cloned. Synced passkeys typically return `0` on every use and should be exempt from monotonicity checks. See [Chapter 5](../chapters/05-passkeys-and-webauthn.md).

*See also:* [W3C WebAuthn §6.1.1 Signature Counter](https://www.w3.org/TR/webauthn-3/#sctn-sign-counter)

---

## SPA

Single Page Application. A web application that loads a single HTML page and dynamically updates the content via JavaScript, without full page reloads. Client-side routing (React Router) changes the URL and renders different components without talking to the server. This project is an SPA — the HTML shell is served once, and all navigation is client-side.

*See also:* [React SPA guide](https://react.dev/learn/start-a-new-react-project)

---

## Step-up Authentication

A security mechanism that requires re-proving identity before performing a sensitive or destructive action, even within an already-authenticated session. Step-up tokens are issued after successful re-authentication and consumed by the sensitive endpoint. This project requires step-up (passkey re-authentication) before account deletion. See [Chapter 10](../chapters/10-step-up-authentication.md).

*See also:* [NIST SP 800-63B: Reauthentication](https://pages.nist.gov/800-63-3/sp800-63b.html)

---

## Token Hash

The SHA-256 hash of a session token. Stored in the database instead of the raw token. This ensures that a database compromise does not yield usable session tokens — the attacker would need to invert SHA-256 (computationally infeasible) to recover the raw tokens. KV entries are also keyed by `session:{tokenHash}`, not by the raw token.

---

## TOTP

Time-Based One-Time Password (RFC 6238). An OTP variant where codes are generated from a shared secret and the current time (typically 30-second windows). Used by "authenticator apps" like Google Authenticator, Authy, and 1Password TOTP. Unlike email OTP, TOTP doesn't require network delivery — the code is computed locally. Listed as future work for this project.

*See also:* [RFC 6238](https://www.rfc-editor.org/rfc/rfc6238)

---

## TTL

Time-To-Live. The duration after which a stored item expires and is automatically deleted. In KV, TTL is specified in seconds when writing an entry (`{ expirationTtl: 300 }`). TTL-based expiration eliminates the need for explicit cleanup of ephemeral data like OTP codes, challenges, and pending sessions. This project uses TTLs ranging from 60 seconds (WebAuthn challenges) to 30 days (trusted session tokens).

---

## User Verification

A WebAuthn requirement for the authenticator to verify the user's identity (biometrics, PIN, etc.) before producing an assertion. Three levels: `required` (must perform UV, reject if not), `preferred` (prefer UV but accept without), `discouraged` (skip UV). This project uses `preferred` for normal sign-in and `required` for step-up authentication (account deletion).

*See also:* [W3C WebAuthn §5.4.6: UserVerificationRequirement](https://www.w3.org/TR/webauthn-3/#dom-userverificationrequirement-required)

---

## Vite

A fast frontend build tool and dev server. Vite uses native ES modules in development (no bundling, instant hot module replacement) and Rollup for production builds. This project uses Vite 6 with the React plugin, Tailwind CSS (PostCSS), and the `vite-plugin-pwa` for Progressive Web App support.

*See also:* [Vite Documentation](https://vitejs.dev/)

## WCAG

Web Content Accessibility Guidelines. A W3C standard for making web content accessible to people with disabilities. The most cited metric is contrast ratio: the ratio of relative luminance between foreground and background colors. WCAG AA requires 4.5:1 for normal text and 3:1 for large text and non-text elements (icons, diagrams). The `Identicon` component enforces the 3:1 non-text threshold (per SC 1.4.11) using the WCAG luminance formula.

*See also:* [WCAG 2.1](https://www.w3.org/TR/WCAG21/)

## WebAuthn

Web Authentication API. A W3C standard (built on FIDO2) that enables web applications to authenticate users using asymmetric cryptography performed on the user's device. The API is `navigator.credentials.create()` (registration) and `navigator.credentials.get()` (authentication). WebAuthn credentials are site-specific (bound to `rpID`), phishing-resistant (the browser enforces origin), and password-free. See [Chapter 5](../chapters/05-passkeys-and-webauthn.md) for the full implementation.

*See also:* [W3C WebAuthn Level 3](https://www.w3.org/TR/webauthn-3/)

## WebSocket

A protocol providing full-duplex, persistent communication over a single TCP connection. Unlike HTTP (request-response), WebSocket allows the server to push data to the client at any time. Initiated with an HTTP upgrade handshake (HTTP 101 status). Used in this project for the number-matching system: trusted devices hold a WebSocket connection open to receive approval push notifications, and new devices hold a connection open to wait for the approval result.

*See also:* [MDN: WebSocket API](https://developer.mozilla.org/en-US/docs/Web/API/WebSockets_API)

## Workers Runtime

The Cloudflare Workers runtime environment. Based on V8 isolates (the same JavaScript engine as Chrome and Node.js), but with a different global API surface than Node.js. Workers expose W3C standards (`fetch`, `crypto.subtle`, `URL`, `Request`, `Response`, `WebSocket`) rather than Node-specific APIs (`fs`, `net`, `path`). The `nodejs_compat` flag extends the runtime with some Node.js compatibility shims.

*See also:* [Cloudflare Workers Runtime APIs](https://developers.cloudflare.com/workers/runtime-apis/)

## Workbox

A set of libraries by Google for service worker-based caching and offline support. `vite-plugin-pwa` generates a Workbox-powered service worker automatically during the build. This project's Workbox config caches all static assets (precaching) and applies a CacheFirst strategy for Google Fonts with a 1-year TTL and a 20-entry max.

*See also:* [Workbox Documentation](https://developer.chrome.com/docs/workbox)

## Wrangler

The Cloudflare Workers CLI tool. Used to develop, test, and deploy Workers. Key commands:
- `wrangler dev` — start a local dev server (using Miniflare)
- `wrangler deploy` — build and deploy the Worker to Cloudflare
- `wrangler d1 execute` — run SQL against a D1 database
- `wrangler kv namespace create` — create a KV namespace
- `wrangler secret put` — store an encrypted secret

*See also:* [Wrangler Documentation](https://developers.cloudflare.com/workers/wrangler/)
