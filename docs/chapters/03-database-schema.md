# Chapter 3 — Database Schema

## What You'll Learn

This chapter documents every table, every column, every index, and every KV key used by the application. It explains why certain data lives in D1 and other data lives in KV, and documents the naming conventions and TTLs for KV keys.

---

## 3.1 The D1 vs KV Split

Before walking through the individual tables, it helps to understand the rule that governs the split.

**D1 (SQLite) holds data that is:**
- Permanent (or whose lifetime is not known in advance)
- Queryable (you need to filter, sort, join, or count it)
- Associated with a user account and needed for the security dashboard

**KV holds data that is:**
- Ephemeral with a known TTL
- Looked up only by a single key (no queries, no joins)
- Frequently accessed for hot-path performance (session auth on every request)

The most illustrative case is sessions. Sessions exist in *both* D1 and KV. The KV entry is used for fast authentication on every request — it is a single key lookup, sub-millisecond. The D1 row is used for the security dashboard (list sessions, show last active time, show device name) and for revocation cleanup. When a session is revoked, both the KV entry and the D1 row must be deleted. This dual-write design is discussed further in [Chapter 7](./07-session-management.md).

## 3.2 D1 Schema

The schema lives in `worker/schema.sql` and is applied with:

```bash
npx wrangler d1 execute varun-portfolio-auth --remote --file=worker/schema.sql
```

### Table: `users`

```sql
CREATE TABLE IF NOT EXISTS users (
  id                        TEXT PRIMARY KEY,
  email                     TEXT UNIQUE NOT NULL,
  created_at                INTEGER NOT NULL,
  nickname                  TEXT,
  frozen_until              INTEGER,
  recovery_rate_limit_until INTEGER
);
```

| Column | Type | Purpose |
|--------|------|---------|
| `id` | TEXT (UUID) | Primary key. Generated with `crypto.randomUUID()`. Never reused. |
| `email` | TEXT | Unique. The only identifier the user provides. Not shown in the UI in full form (masked to `v***@gmail.com`). |
| `created_at` | INTEGER | Unix timestamp in milliseconds. |
| `nickname` | TEXT | Auto-generated on account creation (`swift-fox-42` format). User can edit from the nav avatar menu. |
| `frozen_until` | INTEGER | Unix timestamp in milliseconds. When set and in the future, all auth attempts for this user are rejected with 403. Set when all recovery codes are exhausted through failed attempts. |
| `recovery_rate_limit_until` | INTEGER | Currently unused in code but reserved for rate limiting recovery attempts at the account level. |

**Why a `frozen_until` field instead of a `frozen` boolean?** A boolean requires a manual unfreeze operation. A timestamp automatically expires: the `isUserFrozen` check in `worker/db.js` is simply `Date.now() < user.frozen_until`. A 1-hour freeze resolves itself without operator intervention.

### Table: `passkey_creds`

```sql
CREATE TABLE IF NOT EXISTS passkey_creds (
  id                 TEXT PRIMARY KEY,
  user_id            TEXT NOT NULL REFERENCES users(id),
  public_key         TEXT NOT NULL,
  sign_count         INTEGER NOT NULL DEFAULT 0,
  created_at         INTEGER NOT NULL,
  last_used_at       INTEGER,
  nickname           TEXT,
  transport          TEXT,
  authenticator_type TEXT,
  is_synced          INTEGER NOT NULL DEFAULT 0
);
```

| Column | Type | Purpose |
|--------|------|---------|
| `id` | TEXT | The credential ID from the WebAuthn ceremony. This is a base64url-encoded binary string generated by the authenticator. It is the lookup key during authentication. |
| `user_id` | TEXT | Foreign key to `users.id`. Enforces ownership. Queries are always scoped to a specific user. |
| `public_key` | TEXT | The COSE-encoded public key, stored as base64url via `isoBase64URL.fromBuffer()`. Used to verify authentication assertions. |
| `sign_count` | INTEGER | The authenticator's internal counter. Incremented on each use. Used to detect credential cloning. See [Chapter 5](./05-passkeys-and-webauthn.md) for the full discussion. |
| `created_at` | INTEGER | Unix timestamp in milliseconds. |
| `last_used_at` | INTEGER | Updated on each successful authentication. Used in the security dashboard. |
| `nickname` | TEXT | User-assigned label (e.g., "My iPhone"). Editable from the security settings page. |
| `transport` | TEXT | JSON array of transport hints from the registration response (`['internal']`, `['hybrid']`, etc.). Stored as a string. |
| `authenticator_type` | TEXT | `'platform'` (built-in, like Face ID) or `'cross-platform'` (hardware key or synced credential). |
| `is_synced` | INTEGER (0/1) | Whether the credential is synced across devices (e.g., iCloud Keychain, Google Password Manager). Synced credentials may legitimately have sign_count = 0 on every use. |

**The `is_synced` detection logic:**

```js
// worker/auth/passkey.js lines 109-111
const transports = registrationResponse.response?.transports || [];
const authenticatorType = credential.type || 'platform';
const isSynced = authenticatorType === 'cross-platform' || transports.includes('hybrid');
```

A credential is considered synced if its type is `cross-platform` (meaning it is not a built-in platform authenticator and is likely a cloud-synced passkey) or if its transport includes `hybrid` (the transport mode used by passkeys synced via Bluetooth/QR pairing). This matters for sign count interpretation, discussed in [Chapter 5](./05-passkeys-and-webauthn.md).

### Table: `sessions`

```sql
CREATE TABLE IF NOT EXISTS sessions (
  id             TEXT PRIMARY KEY,
  user_id        TEXT NOT NULL REFERENCES users(id),
  token_hash     TEXT UNIQUE NOT NULL,
  device_name    TEXT,
  user_agent     TEXT,
  ip             TEXT,
  created_at     INTEGER NOT NULL,
  last_active_at INTEGER NOT NULL,
  trusted        INTEGER NOT NULL DEFAULT 0,
  expires_at     INTEGER NOT NULL
);
```

| Column | Type | Purpose |
|--------|------|---------|
| `id` | TEXT (UUID) | Primary key. Used as the session ID in the security dashboard. |
| `user_id` | TEXT | Foreign key. All session queries are scoped by user. |
| `token_hash` | TEXT | SHA-256 hash of the raw session token. The raw token lives only in the browser cookie and in KV. The database never holds the plaintext token. |
| `device_name` | TEXT | Human-readable label inferred from User-Agent or user-provided at trust prompt. |
| `user_agent` | TEXT | Full User-Agent string. Used to detect "known device" during authentication. |
| `ip` | TEXT | Client IP from `CF-Connecting-IP` or `X-Forwarded-For`. Shown on the security dashboard with a "Reveal IP" toggle. |
| `created_at` | INTEGER | When the session was created. |
| `last_active_at` | INTEGER | Updated on every `/me` call (fire-and-forget). Used to show "active 3m ago" in the dashboard. |
| `trusted` | INTEGER (0/1) | Whether the user chose "Trust this device". Trusted sessions last 30 days; untrusted last 24 hours. Also gates number-match approval — only trusted sessions can approve new devices. |
| `expires_at` | INTEGER | Absolute expiration. Used to filter expired sessions from queries. |

**Why store `token_hash` instead of `token`?** If an attacker exfiltrates the sessions table (through a SQL injection or a database backup leak), they gain nothing. Each row contains only the hash of a token. To use a session, you need the raw token (which lives only in the browser cookie). This is equivalent to password hashing applied to session tokens.

### Table: `recovery_codes`

```sql
CREATE TABLE IF NOT EXISTS recovery_codes (
  id         TEXT PRIMARY KEY,
  user_id    TEXT NOT NULL REFERENCES users(id),
  code_hash  TEXT NOT NULL,
  code_salt  TEXT NOT NULL,
  position   INTEGER NOT NULL,
  used       INTEGER NOT NULL DEFAULT 0,
  used_at    INTEGER,
  generation INTEGER NOT NULL
);
```

| Column | Type | Purpose |
|--------|------|---------|
| `id` | TEXT (UUID) | Primary key. |
| `user_id` | TEXT | Foreign key. |
| `code_hash` | TEXT | PBKDF2-SHA256 hash of the recovery code. See [Chapter 9](./09-recovery-system.md). |
| `code_salt` | TEXT | Unique random 16-byte salt per code, hex-encoded. |
| `position` | INTEGER | 1–8. Used to display recovery code status (e.g., "code 3 used"). |
| `used` | INTEGER (0/1) | Whether this code has been consumed. Codes are single-use. |
| `used_at` | INTEGER | Timestamp when the code was used. |
| `generation` | INTEGER | Timestamp used as a generation identifier. When recovery codes are regenerated, all old codes are deleted and a new set is created with a new generation value. This allows detecting "stale" codes in edge cases. |

**Why 8 codes?** Eight is enough to survive losing several codes while still having spares, without being so many that users struggle to store them safely. The position tracking allows the dashboard to show exactly which codes have been consumed without revealing which ones remain valid.

### Table: `security_events`

```sql
CREATE TABLE IF NOT EXISTS security_events (
  id          TEXT PRIMARY KEY,
  user_id     TEXT NOT NULL REFERENCES users(id),
  type        TEXT NOT NULL,
  ip          TEXT,
  user_agent  TEXT,
  device_name TEXT,
  metadata    TEXT,
  created_at  INTEGER NOT NULL
);
```

| Column | Type | Purpose |
|--------|------|---------|
| `type` | TEXT | Event type string. Known types: `login`, `logout`, `passkey_added`, `passkey_removed`, `new_device`, `session_revoked`, `sessions_revoked_all`, `recovery_code_used`, `recovery_code_failed`, `account_recovery`, `account_frozen`, `recovery_codes_regenerated`, `recovery_signin`, `recovery_signin_failed`. |
| `metadata` | TEXT | JSON blob for additional structured data. For example, `passkey_added` includes `{ authenticatorType, isSynced }`. |

Events are append-only. There are no update or delete operations on this table. The dashboard fetches the last 20 events ordered by `created_at DESC`.

## 3.3 D1 Indexes

```sql
CREATE INDEX IF NOT EXISTS idx_passkey_creds_user_id    ON passkey_creds(user_id);
CREATE INDEX IF NOT EXISTS idx_sessions_user_id         ON sessions(user_id);
CREATE INDEX IF NOT EXISTS idx_sessions_token_hash      ON sessions(token_hash);
CREATE INDEX IF NOT EXISTS idx_sessions_expires_at      ON sessions(expires_at);
CREATE INDEX IF NOT EXISTS idx_recovery_codes_user_id   ON recovery_codes(user_id);
CREATE INDEX IF NOT EXISTS idx_security_events_user_id  ON security_events(user_id);
CREATE INDEX IF NOT EXISTS idx_security_events_created  ON security_events(created_at);
```

Every foreign key column has an index. The `sessions.token_hash` index supports the revocation lookup (given a hash, find the session record). The `sessions.expires_at` index supports the "list active sessions" query, which filters by `expires_at > ?`. The `security_events.created_at` index supports the descending sort on the events page.

Note there is no index on `sessions.user_agent`. The `isKnownDevice` check (used during authentication) does a `WHERE user_id = ? AND user_agent = ?` query. Because `user_id` is indexed, SQLite can narrow to the user's sessions first and then scan those rows for a matching user agent. For a user with a small number of active sessions this is fast enough.

## 3.4 KV Key Naming Conventions

All KV keys follow a `prefix:identifier` naming convention. The following table documents every key pattern used in the system.

| Key Pattern | Set by | Value | TTL | Purpose |
|-------------|--------|-------|-----|---------|
| `otp:{email}` | `sendOTP` | `{ code, used }` JSON | 600s (10 min) | Active OTP for email verification |
| `otp_rate:{email}` | `sendOTP` | Count string (e.g. `"2"`) | 600s (10 min) | OTP send rate limit (max 3 per window) |
| `email_verified:{email}` | `verifyOTP` | `"1"` | 300s (5 min) | Gate: email is verified, passkey registration may proceed |
| `reg_challenge:{userId}` | `getRegisterOptions` | Challenge string | 60s | WebAuthn registration challenge |
| `auth_challenge:{userId}` | `getAuthOptions` | Challenge string | 60s | WebAuthn authentication challenge (email flow) |
| `cond_challenge:{condToken}` | `getAuthOptions` | Challenge string | 120s | WebAuthn challenge for conditional mediation (no userId known yet) |
| `session_pending:{token}` | `createPendingSession` | `{ userId, email }` JSON | 300s (5 min) | Pending session awaiting trust prompt completion |
| `session:{tokenHash}` | `finaliseSession` | `{ userId, email, sessionId }` JSON | 86400s or 2592000s | Active session. Keyed by hash, not raw token. |
| `step_up_challenge:{userId}` | `stepUpOptions` | Challenge string | 120s (2 min) | Step-up auth challenge |
| `step_up:{token}` | `stepUpVerify` | `userId` string | 120s (2 min) | Validated step-up token, consumable once |
| `num_match:{approvalToken}` | `verifyAuth` | `{ userId, email, code, approved, denied }` JSON | 300s (5 min) | Pending number match approval |
| `num_match_pending:{tempToken}` | `verifyAuth` | `{ code, approvalToken, userId, email }` JSON | 300s (5 min) | Gate: new device temp token for WS `/wait` endpoint |
| `num_match_for_user:{userId}` | `verifyAuth` | `{ approvalToken, code, userAgent }` JSON | 300s (5 min) | Allows trusted device to find pending approval after DO restart |
| `recovery_otp:{email}` | `recoveryStart` | OTP string | 600s (10 min) | OTP for the full account recovery flow (second factor) |
| `recovery_rate:{email}` | `recoveryStart` | `{ count, first }` JSON | 600s (10 min) | Rate limit for full recovery flow |
| `recovery_signin_rate:{email}` | `recoverySignIn` | `{ count }` JSON | 600s (10 min) | Rate limit for backup sign-in flow |
| `recovery_gate:{recoveryToken}` | `recoveryVerify` | `{ userId, email }` JSON | 300s (5 min) | Gate: identity verified, new passkey registration allowed |

**Design notes:**

The `session:` prefix is critical. The actual KV lookup in `getSession` is:
```js
const tokenHash = await sha256Hex(token);
const raw = await kv.get(`session:${tokenHash}`);
```
The key is `session:` plus the *hash* of the token, not the token itself. This means even the KV namespace does not contain plaintext tokens.

The `cond_challenge:` keys use a short-lived UUID (`condToken`) as the suffix rather than a `userId`, because at the start of conditional mediation, the server does not know which user will authenticate. The `condToken` is returned to the browser, which includes it in the verification call so the server can look up the matching challenge.

## 3.5 D1 Schema — Chat Tables

The RAG chat system (see [Chapter 15](./15-rag-system.md)) adds two tables to `worker/schema.sql`.

### Table: `conversations`

```sql
CREATE TABLE IF NOT EXISTS conversations (
  id         TEXT PRIMARY KEY,
  user_id    TEXT NOT NULL REFERENCES users(id),
  title      TEXT,
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL
);
```

| Column | Type | Purpose |
|--------|------|---------|
| `id` | TEXT (UUID) | Primary key. Generated with `crypto.randomUUID()` at conversation creation. |
| `user_id` | TEXT | Foreign key to `users.id`. All conversation queries are scoped to a specific user — no conversation can be retrieved by a different user, even if the UUID is guessed. |
| `title` | TEXT | Auto-derived from the first 60 characters of the first message, with whitespace collapsed. Never updated after creation. Used in the conversation list sidebar on the `/chat` page. |
| `created_at` | INTEGER | Unix timestamp in milliseconds. Set once at creation. |
| `updated_at` | INTEGER | Unix timestamp in milliseconds. Updated on every `saveMessages` call. Used to sort the conversation list by recency (`ORDER BY updated_at DESC`). |

The conversation list query caps at 50 results (`LIMIT 50`). Older conversations are not shown but still exist in D1 and can be deleted. There is no automatic pruning.

---

### Table: `chat_messages`

```sql
CREATE TABLE IF NOT EXISTS chat_messages (
  id              TEXT PRIMARY KEY,
  conversation_id TEXT NOT NULL REFERENCES conversations(id),
  role            TEXT NOT NULL,
  content         TEXT NOT NULL,
  created_at      INTEGER NOT NULL
);
```

| Column | Type | Purpose |
|--------|------|---------|
| `id` | TEXT (UUID) | Primary key. |
| `conversation_id` | TEXT | Foreign key to `conversations.id`. All message queries are scoped to a conversation. |
| `role` | TEXT | `'user'` or `'assistant'`. Mirrors the chat completion message format used by the Workers AI API. |
| `content` | TEXT | The full message text. For user messages, this is the raw input. For assistant messages, this is the complete response accumulated during streaming via the `onFullText` callback in `transformStream`. |
| `created_at` | INTEGER | Unix timestamp in milliseconds. User and assistant messages for the same turn use `now` and `now + 1` respectively, guaranteeing correct `ORDER BY created_at ASC` ordering even when both rows are inserted in the same millisecond. |

**Why persist after streaming completes, not before?** The assistant message is not known until the stream ends. The `onFullText` callback is invoked by `transformStream` when it receives the `[DONE]` SSE event, at which point `fullText` contains the entire assistant response. Attempting to write incrementally would require either a streaming D1 upsert (not supported) or complex partial-write logic. Writing once at stream end is simpler and correct.

**Conversation history retrieval** for multi-turn context uses:

```js
// worker/chat.js — getConversationHistory
const result = await db
  .prepare(`SELECT role, content FROM chat_messages
            WHERE conversation_id = ?
            ORDER BY created_at DESC LIMIT ${HISTORY_MESSAGES}`)
  .bind(conversationId)
  .all();
return result.results.reverse();
```

`HISTORY_MESSAGES = 10`. The query fetches the 10 most recent messages in reverse order and then reverses the result array in JavaScript to restore chronological order. This is logically equivalent to `ORDER BY created_at ASC LIMIT 10 OFFSET (total - 10)` but avoids a count subquery.

## 3.6 Schema Deletion Order

Because the schema uses foreign key references but no `ON DELETE CASCADE`, the `deleteUser` function in `worker/db.js` deletes child rows first, then the user:

```js
// worker/db.js lines 180-187
export async function deleteUser(db, userId) {
  await db.prepare('DELETE FROM recovery_codes WHERE user_id = ?').bind(userId).run();
  await db.prepare('DELETE FROM security_events WHERE user_id = ?').bind(userId).run();
  await db.prepare('DELETE FROM passkey_creds WHERE user_id = ?').bind(userId).run();
  await db.prepare('DELETE FROM sessions WHERE user_id = ?').bind(userId).run();
  await db.prepare('DELETE FROM users WHERE id = ?').bind(userId).run();
}
```

The order (recovery_codes, security_events, passkey_creds, sessions, users) ensures no referential integrity violation occurs. D1 does not enforce `REFERENCES` as a foreign key constraint by default (SQLite foreign key enforcement requires `PRAGMA foreign_keys = ON`), but following the correct deletion order is good practice.

## Key Takeaways

- D1 stores permanent, queryable data. KV stores ephemeral data with TTLs.
- Sessions exist in both — D1 for the dashboard and revocation lookup, KV for fast per-request authentication.
- The database never stores raw session tokens or raw recovery codes in plaintext. Only hashes.
- KV key naming follows `prefix:identifier` consistently. The `session:` prefix uses the token hash, not the token.
- Synced passkeys are detected by checking for `cross-platform` authenticator type or `hybrid` transport, because they need different sign count handling.
- Chat conversations and messages are stored in D1, not KV. Chat content is permanent and structured (queries, joins, sort by recency) — it belongs in D1. The `updated_at + 1ms` trick ensures correct ordering when user and assistant messages are written in the same millisecond.
