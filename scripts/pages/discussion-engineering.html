Building a Threaded Discussion Board

A full-stack deep dive into building Reddit-style threaded comments on Cloudflare's edge stack — from recursive data structures and React component trees to IntersectionObserver pagination, soft deletes, and production monitoring.

01 Feature Overview

The discussion board is a lightweight, threaded forum embedded in this portfolio site. It follows the same permission model as most developer communities: anyone can read, signed-in users can write.

ActionWhoNotes
Browse topic listPublicInfinite scroll, 50 per page
Read a threadPublicFull comment tree loaded at once
Start a topicSigned-inTitle + body (plain text)
Post a commentSigned-inCan reply to any comment, any depth
Delete a commentOwn author onlySoft delete — tree structure preserved
Pin a topicAdminPinned topics sort first

The feature lives at /discussion for the topic list and /discussion/:id for each thread. The key design goal was no depth limit on replies — any comment can be replied to, and that reply can itself be replied to, forever.

02 Data Architecture

The data lives in Cloudflare D1 — a SQLite-compatible database running at Cloudflare's edge. Two tables: one for topics and one for comments.

Schema

worker/migrations/006-discussion.sql SQL 
CREATE TABLE discussion_topics (
  id            TEXT    PRIMARY KEY,
  title         TEXT    NOT NULL,
  body          TEXT    NOT NULL,
  author_id     TEXT    NOT NULL REFERENCES users(id),
  created_at    TEXT    NOT NULL,  -- ISO 8601 string
  updated_at    TEXT    NOT NULL,
  comment_count INTEGER NOT NULL DEFAULT 0,  -- denormalized
  pinned        INTEGER NOT NULL DEFAULT 0
);

CREATE TABLE discussion_comments (
  id         TEXT    PRIMARY KEY,
  topic_id   TEXT    NOT NULL REFERENCES discussion_topics(id),
  parent_id  TEXT             REFERENCES discussion_comments(id),
  depth      INTEGER NOT NULL DEFAULT 0,
  author_id  TEXT    NOT NULL REFERENCES users(id),
  body       TEXT    NOT NULL,
  created_at TEXT    NOT NULL,
  updated_at TEXT    NOT NULL,
  deleted    INTEGER NOT NULL DEFAULT 0
);

CREATE INDEX idx_comments_topic  ON discussion_comments(topic_id);
CREATE INDEX idx_comments_parent ON discussion_comments(parent_id);

Design decisions

Self-referential parent_id. A comment row references its parent comment in the same table. NULL means it's a root-level comment on the topic. This naturally supports any depth without needing a separate table per level.

Denormalized comment_count. Incrementing a counter column at write time costs one UPDATE, but buying the topic list page an O(n) query instead of an O(n) + n×COUNT aggregation. At the scale of this board (hundreds of topics, not millions), the tradeoff is clear.

Stored depth. The depth column is computed at insert time (parent.depth + 1) and stored. This avoids a recursive query at read time — depth is just a field you read, not something you compute.

Soft delete. When a user deletes their comment, the row isn't removed. Instead deleted = 1 and body = '' are set. The comment node remains in the tree, so existing replies aren't orphaned. The API returns body: null and author: '[deleted]' for deleted entries — the UI renders a greyed-out placeholder.

💡

Why ISO strings for created_at instead of epoch milliseconds? The rest of the codebase (users, sessions) uses epoch integers for D1 queries with numeric range comparisons. For discussion, ISO strings work fine because they're lexicographically sortable — ORDER BY created_at DESC does the right thing without any conversion.

Comment tree structure

In the database, comments are flat rows linked by parent_id. A thread with two root comments, one with a nested reply, looks like this:

── Topic "Thought of the day" │ ├── Comment c1 parent_id=null depth=0 │ "Really resonates with me." │ │ │ └── Comment c2 parent_id=c1 depth=1 │ "Especially the focus part." │ │ │ └── Comment c3 parent_id=c2 depth=2 │ "Agreed, depth 2 reply." │ └── Comment c4 parent_id=null depth=0 "Another top-level comment."

The auth guard pattern

Every write operation runs through a shared guardAuth helper. Sessions are stored in Cloudflare KV, keyed by a hashed token that arrives as an HttpOnly cookie. Invalidating a session is immediate — just delete the KV key.

worker/discussion.js JavaScript 
async function guardAuth(request, env) {
  const session = await getSession(env.KV, request);
  if (!session?.userId) return json({ error: 'Unauthorized' }, 401);
  return session;          // ← { userId, email, ... }
}

export async function createTopic(request, env) {
  const session = await guardAuth(request, env);
  if (session instanceof Response) return session; // 401 early return

  // session.userId is now safely available
  const { title, body } = await request.json().catch(() => ({}));
  ...
}

The limit+1 trick for pagination

Knowing whether "there are more pages" without an extra COUNT query: fetch one row beyond the page size. If you get limit + 1 rows back, more exist. Slice back to limit before returning.

worker/discussion.js — listTopics() JavaScript 
// Fetch one extra to determine whether more pages exist
const rows = await db.prepare(`
  SELECT ... FROM discussion_topics ...
  LIMIT ? OFFSET ?
`).bind(limit + 1, offset).all();

const hasMore = rows.results.length > limit;
const topics  = rows.results.slice(0, limit);

return json({ topics, hasMore });

Atomic comment insert

Adding a comment needs two writes: INSERT the comment row, and INCREMENT comment_count on its topic. These run inside a single D1 batch() call — Cloudflare's equivalent of a transaction for simple cases.

worker/discussion.js — addComment() JavaScript 
await db.batch([
  db.prepare(
    'INSERT INTO discussion_comments (id, topic_id, parent_id, depth, author_id, body, ...) VALUES (?, ?, ?, ?, ?, ?, ?, ?, 0)'
  ).bind(id, topicId, parent_id || null, depth, session.userId, body, now, now),

  db.prepare(
    'UPDATE discussion_topics SET comment_count = comment_count + 1, updated_at = ? WHERE id = ?'
  ).bind(now, topicId),
]);

04 Frontend Architecture

Building the comment tree

The API returns comments as a flat array sorted by created_at ASC. The client reconstructs the tree in O(n) using two passes.

src/pages/DiscussionThread.jsx JavaScript 
function buildTree(comments) {
  const map   = {};
  const roots = [];

  // Pass 1: index every comment by id, add empty replies array
  for (const c of comments) map[c.id] = { ...c, replies: [] };

  // Pass 2: wire each comment into its parent's .replies
  for (const c of comments) {
    if (c.parent_id && map[c.parent_id])
      map[c.parent_id].replies.push(map[c.id]);
    else
      roots.push(map[c.id]);
  }

  return roots; // array of root-level nodes, each with .replies
}

Why two passes? The first pass ensures every node exists in map before the second pass tries to look up parents. Since comments arrive ordered by creation time, a parent always appears before its children in the array — but we can't rely on that for correctness (and a single pass would require that invariant to hold).

The recursive Comment component

Each comment renders itself as a card, then recursively renders its children. No depth limit on the data — every node, at every depth, gets a full <Comment> with its own Reply button and collapse toggle.

src/pages/DiscussionThread.jsx JSX 
function Comment({ comment, topicId, userId, onRefresh, depth = 0 }) {
  const [replying,  setReplying]  = useState(false);
  const [collapsed, setCollapsed] = useState(false);

  return (
    <div className="disc-comment-wrap">
      <div className="disc-comment-card">
        <Avatar name={comment.author} />
        <p>{comment.body}</p>
        <div className="disc-comment-card__actions">
          {userId && <button onClick={() => setReplying(r => !r)}>Reply</button>}
        </div>
        {replying && (
          <InlineReplyForm
            topicId={topicId} parentId={comment.id}
            onPosted={() => { setReplying(false); onRefresh(); }}
          />
        )}
      </div>

      {/* Recursion: each reply renders another <Comment> */}
      {!collapsed && comment.replies?.length > 0 && (
        <div className="disc-comment-replies">
          {comment.replies.map(r => (
            <Comment
              key={r.id} comment={r}
              topicId={topicId} userId={userId}
              onRefresh={onRefresh}
              depth={depth + 1}
            />
          ))}
        </div>
      )}
    </div>
  );
}

Visual indentation is capped at MAX_VISUAL_INDENT = 4 levels to prevent content from squeezing off-screen on mobile. But the recursion itself is uncapped — a depth-10 reply still renders, just without additional left margin after level 4.

Infinite scroll with IntersectionObserver

The topic list loads 50 topics at a time. A zero-height sentinel div sits below the last item. When the browser scrolls it into view (actually 300px before, thanks to rootMargin), the next page is fetched.

⚠️

The stale closure problem. If the IntersectionObserver callback closed over React state variables (offset, hasMore, loading), it would capture their initial values at the time the observer was created and never see updates. The fix: use refs.

src/pages/Discussion.jsx — infinite scroll pattern JavaScript 
// Refs stay current without causing re-renders
const offsetRef  = useRef(0);
const hasMoreRef = useRef(false);
const loadingRef = useRef(false);

const fetchPage = useCallback(async (offset, append) => {
  if (loadingRef.current) return;
  loadingRef.current = true;
  // ... fetch, update offsetRef.current, hasMoreRef.current ...
}, []);  // stable — no deps needed because we read refs, not state

useEffect(() => {
  const observer = new IntersectionObserver(
    (entries) => {
      // reads live ref values — never stale
      if (entries[0].isIntersecting
          && hasMoreRef.current
          && !loadingRef.current) {
        fetchPage(offsetRef.current, true);
      }
    },
    { rootMargin: '300px' }  // start 300px before sentinel enters view
  );
  observer.observe(sentinelRef.current);
  return () => observer.disconnect();
}, [fetchPage]);  // fetchPage is stable, so this runs once

Every time a page loads, offsetRef is updated to offset + next.length and hasMoreRef is set from the API's hasMore flag. The observer always reads the current values, not the ones captured at creation.

05 Visual Design System

Deterministic avatar colors

Each user gets a colored avatar circle. The color is deterministic — the same username always maps to the same color across every page load, every device. No server call needed, no storage required.

src/pages/DiscussionThread.jsx JavaScript 
const AVATAR_COLORS = [
  '#6366f1', '#ec4899', '#f97316', '#10b981',
  '#3b82f6', '#8b5cf6', '#ef4444', '#14b8a6',
];

function avatarColor(name) {
  let h = 0;
  for (let i = 0; i < name.length; i++)
    h = (h * 31 + name.charCodeAt(i)) >>> 0;  // djb2-style, unsigned
  return AVATAR_COLORS[h % AVATAR_COLORS.length];
}

function getInitials(name) {
  const parts = name.split(/[-_\s]/);
  return ((parts[0]?.[0] ?? '') + (parts[1]?.[0] ?? '')).toUpperCase();
}

The >>> 0 coerces to an unsigned 32-bit integer, preventing the hash from going negative and producing a valid modulo result. The hash is a simplified djb2 variant: multiply by 31 and add the character code at each step.

Relative timestamps

"May 26, 2026" tells you less than "2 hours ago" for a recent conversation. Absolute dates only appear when content is older than 30 days.

src/pages/DiscussionThread.jsx JavaScript 
function timeAgo(iso) {
  const diff = Date.now() - new Date(iso).getTime();
  const s = Math.floor(diff / 1000);
  if (s < 60)  return 'just now';
  const m = Math.floor(s / 60);
  if (m < 60)  return `${m}m ago`;
  const h = Math.floor(m / 60);
  if (h < 24)  return `${h}h ago`;
  const d = Math.floor(h / 24);
  if (d < 30)  return `${d}d ago`;
  return new Date(iso).toLocaleDateString('en-US', {
    month: 'short', day: 'numeric', year: 'numeric'
  });
}

06 Security Model

LayerMechanism
Authentication Session token in HttpOnly cookie, hashed key stored in Cloudflare KV. Revocation is instant — delete the KV entry.
Authorization Write endpoints verify session.userId before any DB write. DELETE additionally checks comment.author_id === session.userId.
XSS prevention Plain text only — no Markdown, no HTML. The API stores raw text and the React renderer displays it via text nodes, not dangerouslySetInnerHTML.
SQL injection All D1 queries use .bind() parameterization. Zero string concatenation into SQL.
Soft delete Deleted comments set deleted=1 and clear body=''. Original content is gone; audit trail remains.
CSP connect-src 'self' on all HTML responses prevents browser-side exfiltration of session tokens to third-party endpoints.
Rate limiting Not yet implemented at the application layer. The path forward is Cloudflare's built-in rate limiting rules (per-IP, per-route) applied in wrangler.toml — no Worker code needed.
🔐

Why plain text only? Markdown rendering introduces an XSS surface. A user who posts [click me](javascript:evil()) could execute arbitrary JS if the renderer isn't hardened. Plain text sidesteps the problem entirely — and for a discussion board focused on conversation, Markdown is rarely needed anyway.

Triage signals

The triage tab is built for two concerns:

  • Orphaned topics — a topic older than 24 hours with zero replies is a community health signal. The triage tab surfaces these so the site owner can seed the conversation or archive the topic.
  • Deleted comment log — soft deletes preserve the author and timestamp. The log shows who deleted what and when, in case a moderation review is needed.

10 Key Decisions

Why load all comments at once?

The thread page fetches the full comment tree in a single request. This eliminates N+1 fetches, simplifies client state, and keeps the initial render fast — a single round-trip to the edge. Paginating comments would add cursor management and complicate tree reconstruction across page boundaries without meaningful benefit at current thread depths.

The natural evolution at higher scale is lazy-loading subtrees on demand: load root comments immediately, defer deep replies behind a "load more" control. That keeps the initial payload bounded while preserving the single-request fast path for typical threads.

Why not Markdown?

Every Markdown renderer is a potential XSS vector. The safe path is to sanitize HTML after rendering. The popular sanitizers (DOMPurify, sanitize-html) add bundle weight and need to be kept up to date. For a first version, plain text with white-space: pre-wrap is correct by construction.

Why Cloudflare D1 over a traditional Postgres?

The entire stack runs on Cloudflare: Workers for compute, KV for sessions, D1 for relational data, Vectorize for embeddings. The tradeoffs were deliberate:

  • What D1 gives up: no full-text search, no advisory locks, SQLite's limited ALTER TABLE support, and a single-writer model that makes hot write paths a concern at scale.
  • What D1 buys: zero connection pooling (Workers are stateless — TCP connections between requests don't persist, making traditional Postgres connection limits a real operational problem), colocation with Worker compute at the edge, and no separate hosting provider to operate.
  • Migration trigger: the moment full-text search on thread bodies or high-throughput write bursts become requirements, Postgres via Hyperdrive (Cloudflare's connection pooling proxy) is the right next step — the schema is portable SQLite-compatible SQL.

Why store depth in the comment row?

The alternative is to compute depth at query time with a recursive CTE. In SQLite this works but is expensive and complex. Storing depth as a denormalized column (computed once at INSERT time, queried as a plain integer thereafter) is a textbook denormalization tradeoff — a small write cost for a zero read cost.
